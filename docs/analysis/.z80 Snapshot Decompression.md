- Full ULA contended-memory timing (every read/write to 0x4000-0x7FFF costs extra T-states).
- Exact R-register increment on **every** M1 cycle (including prefixes, HALT, interrupts).
- Precise interrupt timing (50 Hz frame, exact T-state when the interrupt is acknowledged).
- Beeper sound is generated in lock-step with the CPU.

Your port (pure JS on the `fix-contention-R-register-timing` branch) currently has:

- Snapshot parser now correct for this Jetpac v1 file (the marker search + decompress now matches the 16K data + zeros).

**Jetpac (Ultimate, 1983) is extremely timing-sensitive.**
It draws software sprites with tight cycle-counted loops, uses `LD A,R` for quick randomness (enemy spawn positions, movement patterns, and the “rocket appears” sequence), and plays a short beeper tune on start while the main loop waits for the sound to finish before spawning the first enemies and drawing the rocket.

When the emulator is **too fast** (no contention) or the R register drifts even by a few values, the following happens:

- The startup sound plays (or the delay loop finishes at the wrong moment).
- The random seed from R is different → the “spawn enemies / draw rocket” routine either skips or draws to the wrong place.
- The sprite-drawing routines overrun the beam because they were written assuming the exact contended slowdown.

That is why the **player and platform** (drawn in the very first frame after “5”) appear, but everything that is drawn later in the intro sequence is missing.

4. **Interrupt / frame timing**
   Jetpac uses the 50 Hz maskable interrupt to drive the main loop. If the exact T-state when the interrupt is taken is wrong, the whole game can get out of sync.

Deep-dive comparison: your `fix-contention-R-register-timing` branch vs. gasman/jsspeccy3 (gold standard)**

I pulled the exact files from both repos (your branch + the live jsspeccy3 main) and compared the relevant pieces side-by-side. The Jetpac 16K .z80 snapshot is a **v1** file (PC ≠ 0 at offset 6, single 48 KB memory block after the header, compressed, with the classic 00 ED ED 00 terminator).

### 1. Snapshot loading (.z80 parser)

**jsspeccy3 (runtime/snapshot.js)**

- Always treats v1 as 48K.
- Decompresses the entire remaining file (from offset 30) **to exactly 48 KB**, **no marker search**.
- When the input runs out (common in 16K snapshots), `fileBytes[filePtr]` beyond length returns 0 → upper RAM filled with 0 (correct for 16K games).
- Then maps the linear 48 KB block to pages:
  ```js
  memoryPages[5] = first 16 KB   // 0x4000-0x7FFF (screen + lower RAM)
  memoryPages[2] = next 16 KB    // 0x8000-0xBFFF
  memoryPages[0] = last 16 KB    // 0xC000-0xFFFF
  ```

**Your port (src/loader.mjs)**

- Also detects v1 correctly.
- **But** for compressed v1 it **searches for the 00 ED ED 00 terminator** and cuts the source buffer *before* the marker, then decompresses that slice to 48 KB.
- For a true 16K snapshot the marker appears after ~16 KB of data → you decompress only ~16 KB and the rest of `ramImage` stays 0.
- The linear `ramImage` is later handed to the memory system (exact mapping code not in main.mjs, but presumably `ramBanks[0].set(ramImage.subarray(0,0x4000))` etc.).

**Root cause #1 (very likely)**
Your marker search + slice is the difference. On this particular Jetpac file the terminator appears early (or a false-positive 00 ED ED 00 exists in the data), so you only get the first ~16 KB of RAM. The screen (0x4000-0x5AFF) sits entirely inside the first 16 KB bank, but parts of the bitmap/attributes may be truncated or zeroed depending on where the slice happened. jsspeccy3 never slices and always fills the full 48 KB (with 0s when input ends) → full video RAM.

**Fix**
Remove the end-marker search for v1. Do exactly what jsspeccy3 does:

```js
// inside parseZ80, for version === 1 and compressed
const raw = buf.subarray(30);
ramImage = this._z80Decompress(raw, RAM_48K);   // no slicing!
```

(Your `_z80Decompress` already safely stops at `expectedLen` and treats missing input bytes as 0 — perfect.)

### 2. Screen rendering / framebuffer

**jsspeccy3**
Cycle-accurate core builds a special “frame buffer” (border bytes + bitmap+attr pairs) using precomputed screen events. The JS renderer (`runtime/render.js`) just walks that buffer and expands to RGBA.

**Your port (src/frameBuffer.mjs)**You have a very nice hybrid:

- `updateToTstate` (contended mode — the code for `_fillBufferToTstate` was truncated in the fetch but the comment says it logs changes).
- Fallback `generateFromMemory()` which does the classic non-linear address calc **correctly**:

```js
const y0 = y & 0x07;
const y1 = (y & 0x38) >> 3;
const y2 = (y & 0xC0) >> 6;
const bitmapAddr = (y0 << 8) | (y1 << 5) | (y2 << 11) | xByte;   // ← perfect
const attrAddr  = Math.floor(y / 8) * 32 + xByte;
```

This matches the canonical formula and the one I gave earlier. The `FrameRenderer` is almost identical to jsspeccy3’s (palette, flash handling, double pixels for borders, `putImageData`).

**So the address formula is NOT the bug here.**

The only way the screen can be partial is if the **source memory** (the bitmap/attr views) is partial after loading the snapshot.
